# Coach Weekly Check-In Risk Plan

## 1. Executive summary

`weeklyCheckInRow` is visible near the bottom of `CoachTabView` and currently still feels older than the newly accepted early-state calibration modules. The accepted `earlyStateCard` and `calibrationChecklist` now read as more intentional, while Weekly Check-In still uses the older local grouped card treatment.

The row is action-adjacent. In some states it is passive/locked, but in ready or established states it can call `vm.generateWeeklyReview()` and set `showWeeklyReview = true`, which routes into the `WeeklyCheckInView` sheet when `vm.weeklyReview` exists.

This pass makes no Swift changes. It creates only a docs plan and appends a migration-log entry.

Future implementation should start with the passive early-stage / not-ready shell if safe. That is the narrowest visual target because it should not call review generation, should not open a sheet, and can be improved without touching review data, review actions, or plan-changing behavior.

Figma note: @Figma was used read-only for bounded inspiration. The scan looked only for broad pattern categories such as dark dashboards, card/list components, progress/step motifs, and coach-related surfaces. No Figma assets, layouts, text, or proprietary details should be copied 1:1.

## 2. Current implementation inventory

Target file:

- `ios/STRQ/Views/CoachTabView.swift`

Relevant local state and sheet route:

- `@State private var showWeeklyReview: Bool = false`
- `.sheet(isPresented: $showWeeklyReview) { if let review = vm.weeklyReview { WeeklyCheckInView(vm: vm, review: review) } }`

Relevant helpers and rows:

- `private var weeklyCheckInRow`
- `private func weeklyReviewLabel(subtitle:ready:)`
- `private func planQualityRow(_ quality: PlanQualityScore) -> some View`

Inputs read by `weeklyCheckInRow`:

- `vm.isEarlyStage`
- `vm.planQuality`
- `vm.isWeeklyReviewReady`
- `vm.sessionsUntilReviewReady`

Actions and route triggers:

- Ready branch: `vm.generateWeeklyReview()` then `showWeeklyReview = true`
- Established fallback branch: `vm.generateWeeklyReview()` then `showWeeklyReview = true`
- Passive early-stage branch: no `Button`, no call to `generateWeeklyReview`, no sheet trigger
- Sheet guard: `if let review = vm.weeklyReview`
- Sheet destination: `WeeklyCheckInView(vm: vm, review: review)`

Current branch order:

1. If `!vm.isEarlyStage, let quality = vm.planQuality`, render `planQualityRow(quality)` above the check-in entry.
2. If `vm.isWeeklyReviewReady`, render a `Button` that generates a review and opens the sheet.
3. Else if `vm.isEarlyStage`, render passive `weeklyReviewLabel(..., ready: false)` with `.opacity(0.7)`.
4. Else render an established fallback `Button` that also generates a review and opens the sheet.

Important behavior nuance:

- `vm.isWeeklyReviewReady` is checked before `vm.isEarlyStage`. If an early-stage user ever satisfies the weekly-ready condition and the review has not been dismissed, the current code uses the ready button branch, not the passive early-stage branch. The first implementation pass should not "fix" this unless explicitly approved.

View-model behavior mapped from `AppViewModel`:

- `vm.isEarlyStage` is `dataMaturityTier < .established`.
- `dataMaturityTier` is `.fresh` at 0 completed workouts, `.firstSession` at 1, `.earlyWeek` through 3, and `.established` after that.
- `vm.isWeeklyReviewReady` returns false when `weeklyReviewDismissed` is true; otherwise it checks whether completed sessions in the last 7 days are at least `max(1, profile.daysPerWeek - 1)`.
- `vm.sessionsUntilReviewReady` is `max(0, max(1, profile.daysPerWeek - 1) - weeklyStats.sessions)`.
- `vm.generateWeeklyReview()` assigns `weeklyReview = reviewGenerator.generate(...)`.
- `vm.openWeeklyReview()` exists and tracks `.weekly_review_opened`, but `CoachTabView` does not call it. Do not switch to it unless analytics changes are explicitly scoped.

## 5. Current visual diagnosis

The current Weekly Check-In entry still looks like an older Forge/local card. It uses `Color(.secondarySystemGroupedBackground)`, `STRQBrand.cardBorder`, a compact HStack row, and `STRQBrand.steelGradient` in the icon well. That treatment is serviceable, but it now feels less intentional than the accepted early-state modules above it.

Visible orange/accent debt is conditional rather than always present in the weekly label itself. The weekly row uses steel, but the adjacent `planQualityRow` can resolve `quality.overallColor` to orange through `ForgeTheme.color(for:)` when the quality score is low. Broader CoachTab still has known warm/accent debt elsewhere, but this plan should not attempt a broad CoachTab accent migration.

The row currently feels ambiguous. In passive early-stage states, opacity reduction makes it feel unavailable, but the card still resembles a tappable row. In ready and established states, it becomes a CTA-like gateway because it is wrapped in a `Button` and shows a chevron. Established-not-ready also renders the same ready-looking button, which makes visual treatment especially sensitive.

Compared with the new `earlyStateCard` and `calibrationChecklist`, Weekly Check-In lacks a distinct product role. The accepted early modules speak to calibration and what STRQ is learning. The weekly row should become the weekly reflection loop, not another generic card.

It should not copy Profile or NotificationSettings row patterns. Those surfaces are settings and controls; Weekly Check-In lives inside CoachTab, where the product job is coaching interpretation, not preference management or reminder configuration.

It should not copy the calibrationChecklist exactly. Calibration is about first signals and data maturity; Weekly Check-In is about turning a week of training into a readable review. Reusing the same checklist/progress language would flatten two different product meanings.

## 6. Product goal for Weekly Check-In entry

Define Weekly Check-In as:

- a weekly reflection loop
- a progress recap gateway
- a coach review invitation
- a compact signal that the user's week is becoming readable

It is not:

- a workout CTA
- a subscription card
- a generic settings row
- a reward badge
- a plan mutation surface by itself

The entry should feel like: "your week is becoming readable." Passive states should make the user feel that STRQ is accumulating enough training evidence. Ready states should feel like an invitation to review, not a loud demand to act.

## 8. Visual redesign direction

The Weekly Check-In entry should be distinct from Profile and NotificationSettings. It can share STRQ's calm dark foundation, but it should not read like a setting row, reminder toggle, or account/status control.

It should also be distinct from `calibrationChecklist`. A useful direction is a compact "weekly review gateway" with a timeline, book, field note, report, or quiet recap motif. It should imply that the week is being assembled into a coach-readable summary.

Recommended visual direction:

- calm dark/card shell with better depth than `secondarySystemGroupedBackground`
- compact weekly review gateway, not a full module rewrite
- subtle review/navy or neutral graphite accent
- optional locked/ready status treatment in later approved passes
- passive state that looks clearly unavailable/upcoming without feeling disabled or dead
- ready state that looks tappable without becoming aggressive
- iconography that suggests review, report, timeline, or notes
- hierarchy that keeps existing title/subtitle copy readable on small iPhones

Avoid:

- copying Profile row anatomy
- copying NotificationSettings reminder card anatomy
- copying `calibrationChecklist` progress dots or exact shell
- Pro violet
- orange CTA gradients
- new action language
- new review generation logic
- pairing row redesign with `WeeklyCheckInView` redesign
- broad CoachTab visual replacement in the same pass

Figma-inspired category note:

- The read-only Figma scan supports using broad categories such as dark dashboard cards, compact app cards, list-item hierarchy, progress/step cues, and coach-summary surfaces as inspiration only. STRQ should translate those categories into its own weekly reflection role rather than copying source screens.

## 9. State coverage requirements

Any implementation pass must cover:

- early-stage not ready
- early-stage almost ready with 1 session remaining
- early-stage 0 sessions remaining but not ready because dismissal or guard state blocks readiness
- early-stage weekly-ready branch, because current branch order can make it actionable
- established ready
- established not ready
- `planQualityRow` visible
- `planQualityRow` absent
- `weeklyReview` generated and sheet opens
- `weeklyReview` missing and sheet guard prevents destination rendering
- navigation back/dismiss from `WeeklyCheckInView`
- small iPhone viewport
- large iPhone viewport
- no accidental review generation in passive state
- no accidental sheet open in passive state
- no copy wrapping/clipping in long sessions-until-ready copy
- no visual regression to orange CTA or Pro-violet treatment

Rork QA should not test plan-changing weekly review actions unless the owner approves disposable/safe data. The row pass should verify route opening and dismissing, not apply review actions.

## 10. Risk rating

| Risk area | Rating | Reason |
|---|---:|---|
| Behavior risk | Medium/High | The passive branch is safe, but adjacent ready/fallback branches generate data and open a sheet. |
| Weekly review/action risk | High | The row leads to `WeeklyCheckInView`, where suggested actions can call `vm.applyReviewAction(_:)` and mutate plan state. |
| Product trust risk | High | Locked, almost-ready, ready, and fallback states must not mislead users about what STRQ knows. |
| Visual risk | Medium | The current row is visually older, and the established fallback looks ready even when `isWeeklyReviewReady` is false. |
| Owner approval need | High | Any Swift implementation should be owner-approved with exact state screenshots and no sheet redesign bundled. |

Overall risk: Low for this docs-only pass; Medium/High for a passive shell implementation; High for ready/action branches or sheet work.
